# Система оплаты и пополнения баланса

## 📋 Введение

Данный документ описывает бизнес-процессы и технические потоки системы оплаты в медицинском сервисе MedVito. Система обеспечивает безопасное пополнение баланса пользователей через интеграцию с платёжным провайдером Т-Банк.

### Цели системы
- 🎯 Обеспечить удобное пополнение баланса пользователями
- 💰 Автоматизировать учёт платежей и начисление средств
- 🔒 Гарантировать безопасность финансовых операций
- 📊 Предоставить полную историю транзакций

### Ключевые принципы
- **Идемпотентность**: повторные операции не приводят к дублированию
- **Безопасность**: все операции защищены криптографическими подписями
- **Прозрачность**: полное логирование всех действий с балансом
- **Надёжность**: транзакционная целостность данных

## 🏢 Бизнес-контекст

### Участники процесса

| Роль | Описание | Действия |
|------|----------|----------|
| **Пользователь** | Клиент платформы MedVito | Инициирует пополнение, оплачивает, проверяет баланс |
| **Платформа MedVito** | Медицинский сервис | Создаёт платёж, обрабатывает результат, начисляет баланс |
| **Т-Банк** | Платёжный провайдер | Обрабатывает платёж, отправляет уведомления |
| **Пользовательский банк** | Банк-эмитент карты | Списывает средства со счёта пользователя |

### Бизнес-модель

```
Пользователь → Пополняет баланс → Использует баланс на услуги
     ↓                                      ↓
Т-Банк (комиссия)                    MedVito (доход)
```

## 🔄 Бизнес-процессы

## Процесс 1: Пополнение баланса пользователем

### Описание
Пользователь хочет пополнить свой баланс на платформе для последующей оплаты медицинских услуг, консультаций или покупки товаров.

### Участники
- Пользователь (инициатор)
- Система MedVito
- Т-Банк (платёжный шлюз)

### Предусловия
- Пользователь авторизован в системе
- Платёжная интеграция настроена и активна
- Минимальная сумма пополнения: 1 рубль

### Шаги процесса

```
┌─────────────────────────────────────────────────────────────────┐
│                   ПОПОЛНЕНИЕ БАЛАНСА                             │
└─────────────────────────────────────────────────────────────────┘

1. ИНИЦИАЦИЯ ПЛАТЕЖА
   ├─ Пользователь заходит в личный кабинет
   ├─ Открывает раздел "Баланс" или "Пополнить"
   ├─ Указывает сумму пополнения (мин. 1₽)
   └─ Нажимает "Пополнить баланс"

2. СОЗДАНИЕ ПЛАТЁЖНОЙ СЕССИИ
   ├─ Система генерирует уникальный заказ (OrderID)
   ├─ Отправляет запрос в Т-Банк для создания платежа
   ├─ Получает ссылку на страницу оплаты
   └─ Сохраняет платёж в базе со статусом "Ожидает оплаты"

3. ОПЛАТА
   ├─ Пользователь перенаправляется на защищённую страницу Т-Банк
   ├─ Вводит данные банковской карты
   ├─ Подтверждает платёж (может потребоваться SMS-код)
   └─ Т-Банк обрабатывает транзакцию

4. УВЕДОМЛЕНИЕ О РЕЗУЛЬТАТЕ
   ├─ Т-Банк отправляет webhook в систему MedVito
   ├─ Система проверяет подлинность уведомления
   └─ Обрабатывает результат:
       ├─ УСПЕХ → начисляет баланс, статус "Завершён"
       └─ ОТКАЗ → статус "Отклонён", баланс не начисляется

5. ЗАВЕРШЕНИЕ
   ├─ Пользователь видит обновлённый баланс
   ├─ Получает подтверждение на email (опционально)
   └─ Операция записывается в историю транзакций
```

### Результаты

✅ **Успешный сценарий:**
- Баланс пользователя увеличен на сумму платежа
- Платёж в статусе "Завершён" (COMPLETED)
- Создана запись в журнале операций (Log)
- Пользователь может использовать средства

❌ **Неуспешный сценарий:**
- Баланс не изменён
- Платёж в статусе "Отклонён" (FAILED)
- Причина отказа зафиксирована
- Пользователь может повторить попытку

### Временные характеристики
- Создание платежа: < 2 секунд
- Оплата картой: 10-60 секунд (зависит от пользователя)
- Обработка webhook: < 1 секунды
- Отображение баланса: мгновенно после webhook

## Процесс 2: Обработка платёжных уведомлений (Webhook)

### Описание
Автоматический процесс обработки уведомлений от Т-Банк о статусе платежа. Критически важен для начисления баланса.

### Типы уведомлений

| Статус | Значение | Действие системы |
|--------|----------|------------------|
| **CONFIRMED** | Платёж успешно завершён | ✅ Начислить баланс |
| **REJECTED** | Платёж отклонён банком | ❌ Пометить как отклонённый |
| **CANCELED** | Платёж отменён пользователем | ❌ Пометить как отклонённый |
| **AUTHORIZED** | Только авторизация | ⏳ Ожидать подтверждения |
| **NEW** | Создан, не оплачен | ⏳ Ожидать действий |

### Бизнес-правила

#### 1. Идемпотентность
```
Правило: Повторное уведомление не должно привести к двойному начислению
Реализация: Проверка текущего статуса платежа перед обработкой
Пример: Если платёж уже в статусе "Завершён" → пропустить обработку
```

#### 2. Безопасность
```
Правило: Обрабатывать только подлинные уведомления от Т-Банк
Реализация: Верификация криптографической подписи (Token)
Пример: Если подпись не совпадает → отклонить уведомление
```

#### 3. Атомарность
```
Правило: Баланс и статус обновляются одновременно или не обновляются вообще
Реализация: Транзакция базы данных с блокировкой записей
Пример: При ошибке логирования → откатить начисление баланса
```

### Процесс обработки успешного платежа

```
┌─────────────────────────────────────────────────────────────────┐
│           ОБРАБОТКА УСПЕШНОГО ПЛАТЕЖА (CONFIRMED)                │
└─────────────────────────────────────────────────────────────────┘

1. ПОЛУЧЕНИЕ УВЕДОМЛЕНИЯ
   └─ Т-Банк → POST /payment/notification

2. ВАЛИДАЦИЯ
   ├─ Проверка формата JSON
   ├─ Верификация подписи Token
   └─ ✅ Подпись валидна → продолжить
       ❌ Подпись невалидна → отклонить

3. ПРОВЕРКА СТАТУСА
   ├─ Найти платёж по PaymentID
   ├─ Проверить текущий статус
   └─ Если статус = "Ожидает оплаты" (PENDING):
       └─ Продолжить обработку
       Иначе:
       └─ Вернуть успех (уже обработан)

4. НАЧИСЛЕНИЕ БАЛАНСА (ТРАНЗАКЦИЯ)
   ├─ Начать транзакцию БД
   ├─ Заблокировать записи Payment и User
   ├─ Обновить статус платежа → "Завершён"
   ├─ Увеличить баланс пользователя
   ├─ Создать запись в журнале операций
   └─ Зафиксировать транзакцию

5. ОТВЕТ Т-БАНК
   └─ {"success": true, "message": "Баланс успешно пополнен"}
```

### Обработка ошибок

| Ошибка | Причина | Действие |
|--------|---------|----------|
| Неверная подпись | Подделка или технический сбой | Отклонить, залогировать |
| Платёж не найден | Несуществующий PaymentID | Вернуть ошибку |
| Ошибка БД | Технический сбой | Откатить транзакцию, повтор от Т-Банк |
| Timeout | Перегрузка системы | Т-Банк повторит запрос |

## Процесс 3: Просмотр истории платежей

### Описание
Пользователь может в любое время просмотреть историю всех своих пополнений баланса.

### Отображаемая информация

```
┌─────────────────────────────────────────────────────────────────┐
│                    ИСТОРИЯ ПЛАТЕЖЕЙ                              │
└─────────────────────────────────────────────────────────────────┘

Для каждой транзакции:
├─ Дата и время создания
├─ Сумма платежа
├─ Текущий статус:
│  ├─ 🟡 Ожидает оплаты (PENDING)
│  ├─ 🟢 Завершён (COMPLETED)
│  └─ 🔴 Отклонён (FAILED)
├─ Номер заказа
└─ Ссылка на оплату (если статус PENDING)
```

### Бизнес-значение
- **Прозрачность**: пользователь видит все операции
- **Контроль**: возможность отследить незавершённые платежи
- **Поддержка**: упрощает решение спорных вопросов

## Процесс 4: Проверка статуса платежа

### Описание
Проактивная проверка текущего статуса конкретного платежа в системе Т-Банк.

### Когда используется
- Пользователь не уверен, прошёл ли платёж
- Webhook не поступил в течение длительного времени
- Диагностика проблем службой поддержки

### Процесс
```
Пользователь → Запрос статуса → Система → Запрос в Т-Банк → Актуальный статус
                                    ↓
                         Синхронизация при расхождении
```

## 📊 Статусная модель платежа

### Диаграмма переходов статусов

```
                    [Создание платежа]
                           ↓
                    ┌──────────────┐
                    │   PENDING    │ ← Ожидает оплаты
                    │ (Ожидает)    │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
       [Успешная     [Отклонён]   [Отменён]
        оплата]        банком]   пользователем]
              ↓            ↓            ↓
      ┌──────────┐  ┌──────────┐ ┌──────────┐
      │COMPLETED │  │  FAILED  │ │  FAILED  │
      │(Завершён)│  │(Отклонён)│ │(Отклонён)│
      └──────────┘  └──────────┘ └──────────┘
```

### Описание статусов

| Статус | Когда устанавливается | Значение для бизнеса | Действия пользователя |
|--------|----------------------|---------------------|---------------------|
| **PENDING** | При создании платежа | Деньги в процессе | Ожидать или оплатить по ссылке |
| **COMPLETED** | После успешной оплаты | Баланс начислен | Можно использовать средства |
| **FAILED** | После отказа/отмены | Баланс не начислен | Можно создать новый платёж |

## 🔐 Безопасность и защита

### Механизмы защиты

#### 1. Криптографическая подпись
```
Все запросы к Т-Банк и от Т-Банк подписываются SHA256 токеном
Предотвращает: подделку запросов, MITM атаки
```

#### 2. Верификация webhook
```
Каждое уведомление от Т-Банк проверяется по секретному ключу
Предотвращает: фальшивые начисления баланса
```

#### 3. Транзакционная целостность
```
Баланс и статус обновляются атомарно в одной транзакции
Предотвращает: рассинхронизацию данных, потерю средств
```

#### 4. Идемпотентность
```
Повторная обработка не меняет результат
Предотвращает: двойное начисление при повторных webhook
```

#### 5. Аудит и логирование
```
Все операции с балансом логируются с полными деталями
Обеспечивает: прозрачность, возможность расследования
```

### Хранение секретов
- ✅ Ключи Т-Банк хранятся в переменных окружения
- ✅ Не попадают в логи и код
- ✅ Доступ только серверному приложению

## 💼 Бизнес-метрики

### Ключевые показатели (KPI)

| Метрика | Формула | Значение для бизнеса |
|---------|---------|---------------------|
| **Конверсия платежей** | Completed / Created × 100% | Эффективность платёжного процесса |
| **Средний чек** | Σ Amount / Count(Completed) | Типичная сумма пополнения |
| **Отказы** | Failed / Created × 100% | Проблемы с оплатой |
| **Время обработки** | Time(Webhook) - Time(Created) | Скорость начисления |
| **Повторные платежи** | Count(User payments > 1) | Лояльность пользователей |

### Мониторинг

```
🟢 Норма: Конверсия > 80%, отказы < 10%
🟡 Внимание: Конверсия 60-80%, отказы 10-20%
🔴 Проблема: Конверсия < 60%, отказы > 20%
```

### События для аналитики

- ✅ Создан платёж (amount, userId)
- ✅ Успешная оплата (amount, время обработки)
- ❌ Отклонён платёж (причина)
- 👁️ Просмотр истории платежей
- 🔍 Проверка статуса платежа

## 🚨 Обработка нестандартных ситуаций

### Сценарий 1: Не пришёл webhook

**Проблема**: Пользователь оплатил, но баланс не начислен

**Причины**:
- Сетевые проблемы между Т-Банк и MedVito
- Временная недоступность сервера
- Технический сбой в Т-Банк

**Решение**:
1. Т-Банк автоматически повторяет webhook несколько раз
2. Пользователь может проверить статус вручную
3. Поддержка может запросить статус через API GetState

---

### Сценарий 2: Двойной webhook

**Проблема**: Т-Банк отправил уведомление дважды

**Защита**: Идемпотентность
- Первый webhook: обрабатывается, баланс начисляется
- Второй webhook: статус уже COMPLETED → пропускается

**Результат**: Баланс начислен только один раз ✅

---

### Сценарий 3: Частичный сбой при обработке

**Проблема**: Ошибка в процессе транзакции

**Защита**: Транзакция БД
- Если любой шаг не удался → откат всех изменений
- Т-Банк повторит webhook → обработка произойдёт

**Результат**: Либо полностью успешно, либо ничего ✅

---

### Сценарий 4: "Зависший" платёж

**Проблема**: Платёж в статусе PENDING более 24 часов

**Причины**:
- Пользователь не завершил оплату
- Техническая проблема

**Действия**:
- Автоматический мониторинг старых PENDING
- Проверка статуса через GetState API
- Уведомление пользователя (опционально)

## 🎯 Рекомендации и Best Practices

### Для бизнеса

1. **Мониторинг конверсии**
   - Отслеживайте процент успешных платежей
   - Анализируйте причины отказов
   - Оптимизируйте UX процесса оплаты

2. **Работа с отказами**
   - Предлагайте альтернативные способы оплаты
   - Уведомляйте об ошибках понятным языком
   - Предоставляйте помощь техподдержки

3. **Стимулирование пополнений**
   - Бонусы за крупные пополнения
   - Напоминания о низком балансе
   - Акции и специальные предложения

### Для разработки

1. **Надёжность**
   - Используйте транзакции для критичных операций
   - Реализуйте идемпотентность всех операций
   - Логируйте все финансовые действия

2. **Безопасность**
   - Всегда верифицируйте подписи
   - Храните секреты безопасно
   - Регулярно проверяйте логи на аномалии

3. **Производительность**
   - Оптимизируйте запросы к БД
   - Используйте connection pooling
   - Мониторьте время ответа API

### Для поддержки

1. **Диагностика**
   - Проверяйте статус в истории платежей
   - Используйте Check Status API для актуальных данных
   - Анализируйте логи операций (таблица Log)

2. **Разрешение споров**
   - История платежей содержит все детали
   - Логи показывают точное время операций
   - PaymentID для связи с Т-Банк

### Модель данных

**Таблица Payment** - хранит информацию о всех платежах:
- Уникальные идентификаторы (orderId, paymentId)
- Связь с пользователем (userId)
- Сумма и статус платежа
- Временные метки создания и обновления

**Таблица User** - содержит балансы пользователей:
- Основной баланс (balance)
- Бонусный баланс (bonusBalance)

**Таблица Log** - аудит финансовых операций:
- История всех изменений баланса
- Связь с пользователем и описание действия

## ⚙️ Конфигурация и интеграция

### Настройка интеграции с Т-Банк

Для работы системы необходимо получить в личном кабинете Т-Банк:
1. **Terminal Key** - публичный идентификатор терминала
2. **Secret Key** - секретный ключ для подписи запросов

Конфигурация через переменные окружения:
```bash
TINKOFF_TERMINAL_KEY=your_terminal_key
TINKOFF_SECRET_KEY=your_secret_key
```

### Настройка webhook в Т-Банк

В личном кабинете Т-Банк необходимо указать URL для уведомлений:
```
https://your-domain.com/payment/notification
```

Т-Банк будет отправлять на этот адрес POST-запросы с результатами платежей.

## 🧪 Тестирование

### Тестовый режим

Т-Банк предоставляет тестовую среду для отладки интеграции:
- Отдельные тестовые ключи (Terminal Key и Secret Key)
- Тестовые номера карт для проверки различных сценариев
- Параметр `test: 1` в ответах API означает тестовую транзакцию

### Тестовые сценарии

**Успешная оплата**:
1. Создать платёж на любую сумму ≥ 1₽
2. Перейти по полученной ссылке
3. Использовать тестовую карту (из документации Т-Банк)
4. Подтвердить оплату
5. Проверить начисление баланса

**Отклонённая оплата**:
1. Использовать специальную тестовую карту для отказа
2. Убедиться, что статус изменился на FAILED
3. Баланс не должен измениться

**Отменённая оплата**:
1. Создать платёж
2. Закрыть страницу оплаты без завершения
3. Статус остаётся PENDING (можно повторить попытку)

## 🔧 Техническая поддержка

### Частые проблемы и решения

**Проблема**: "Платёжная система не настроена"
- **Причина**: Отсутствуют переменные окружения
- **Решение**: Проверить TINKOFF_TERMINAL_KEY и TINKOFF_SECRET_KEY

**Проблема**: "Неверная подпись уведомления"
- **Причина**: Неправильный Secret Key или изменён формат данных
- **Решение**: Проверить актуальность Secret Key в конфигурации

**Проблема**: Баланс не начисляется после оплаты
- **Причина**: Webhook не поступает на сервер
- **Решение**: 
  1. Проверить доступность /payment/notification извне
  2. Проверить правильность URL в личном кабинете Т-Банк
  3. Использовать Check Status API для синхронизации

**Проблема**: Долгое ожидание статуса PENDING
- **Причина**: Пользователь не завершил оплату
- **Решение**: Пользователь может вернуться по ссылке paymentUrl

### Логи для диагностики

При обращении в поддержку полезны следующие данные:
- PaymentID или OrderID платежа
- Email пользователя
- Время создания платежа
- Текущий статус из /payment/history
- Сообщения об ошибках (если были)

## 📊 Аналитика и отчётность

### Рекомендуемые дашборды

**Операционный дашборд** (для ежедневного мониторинга):
- Количество платежей за период (час/день)
- Конверсия платежей (%)
- Средний чек
- Сумма успешных пополнений
- Количество ошибок и отказов

**Бизнес-дашборд** (для руководства):
- Динамика пополнений (тренды)
- Распределение по суммам
- Повторные пополнения пользователей
- Lifetime Value пользователя
- ROI от различных каналов привлечения

**Технический дашборд** (для DevOps):
- Время обработки webhook (p50, p95, p99)
- Количество ошибок интеграции
- Доступность внешних API
- Зависшие PENDING платежи

### SQL запросы для отчётов

**Конверсия за последние 30 дней**:
```sql
SELECT 
  COUNT(*) FILTER (WHERE status = 'COMPLETED') * 100.0 / COUNT(*) as conversion,
  COUNT(*) as total_payments,
  COUNT(*) FILTER (WHERE status = 'COMPLETED') as successful,
  COUNT(*) FILTER (WHERE status = 'FAILED') as failed,
  COUNT(*) FILTER (WHERE status = 'PENDING') as pending
FROM "Payment"
WHERE "createdAt" >= NOW() - INTERVAL '30 days';
```

**Средний чек по периодам**:
```sql
SELECT 
  DATE_TRUNC('day', "createdAt") as date,
  COUNT(*) as count,
  AVG(amount) as avg_amount,
  SUM(amount) as total_amount
FROM "Payment"
WHERE status = 'COMPLETED'
  AND "createdAt" >= NOW() - INTERVAL '30 days'
GROUP BY DATE_TRUNC('day', "createdAt")
ORDER BY date DESC;
```

**Топ пользователей по количеству пополнений**:
```sql
SELECT 
  "userId",
  u.email,
  COUNT(*) as payment_count,
  SUM(p.amount) as total_amount
FROM "Payment" p
JOIN "User" u ON u.id = p."userId"
WHERE p.status = 'COMPLETED'
GROUP BY "userId", u.email
ORDER BY payment_count DESC
LIMIT 20;
```

## 📖 Глоссарий

| Термин | Определение |
|--------|-------------|
| **OrderID** | Уникальный идентификатор заказа в системе MedVito (формат: userId-timestamp) |
| **PaymentID** | Уникальный идентификатор платежа в системе Т-Банк |
| **Webhook** | Автоматическое HTTP-уведомление от Т-Банк о статусе платежа |
| **Token** | Криптографическая подпись запроса (SHA256), защищает от подделки |
| **Идемпотентность** | Свойство операции давать одинаковый результат при повторном выполнении |
| **Транзакция** | Группа операций БД, которые выполняются атомарно (все или ничего) |
| **Конверсия** | Процент успешных платежей от общего числа созданных |
| **Terminal Key** | Публичный идентификатор терминала Т-Банк |
| **Secret Key** | Секретный ключ для подписи запросов к Т-Банк |
| **Баланс** | Доступные средства пользователя на платформе |
| **Бонусный баланс** | Дополнительные средства (акции, кешбэк), учитываются отдельно |
